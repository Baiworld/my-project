## 1. 项目概述

### 1.1 项目背景

当前音乐与视频平台面临两大核心挑战：（1）**新用户冷启动**——缺少历史行为数据，推荐质量差，用户留存率低；（2）**多模态内容混合推荐**——音乐与视频分属不同推荐链路，无法形成跨模态协同效应，用户从一个内容类型切换到另一个时体验断裂。

本项目构建一套基于 Spark 大数据的用户行为实时分析与混合推荐系统，通过 Kafka 实时接入用户行为数据，Spark Streaming 进行实时特征分析与聚合，SparkSQL 驱动离线模型训练与批量推荐，结合 Python Flask + Vue 提供业务可视化管理平台，实现"实时分析 → 离线训练 → 混合推荐 → 效果展示"的完整闭环。

### 1.2 项目目标

| 目标维度 | 具体描述 |
|---------|---------|
| 冷启动解决 | 新用户在 5 次交互内获得个性化推荐，冷启动阶段 CTR 提升 30% |
| 混合推荐 | 同一 feed 流内混合推荐音乐与视频内容，跨模态点击率提升 20% |
| 实时响应 | 用户行为发生后 3 秒内完成特征更新，推荐结果实时调整 |
| 离线深度 | 每 6 小时完成一次全量用户画像更新与模型重训练 |
| 可视化管理 | 提供数据大屏实时监控推荐效果，支持运营人员条件查询与干预 |

### 1.3 适用范围

- 平台新注册用户（0-50 条行为记录）的冷启动推荐
- 存量用户（>50 条行为记录）的音乐+视频混合推荐
- 运营人员的数据监控、推荐效果评估、手动干预

## 3. 功能需求

### 3.1 数据接入模块（Kafka → Spark Streaming）

**FR-01 Kafka 实时数据源接入**
- 系统应从 Kafka 消费以下 Topic：
  - `user_behavior`：用户行为日志（播放、点赞、收藏、评论、跳过、完播、分享）
  - `content_metadata`：内容元数据（音乐：歌手、风格、语种、时长、BPM；视频：分类、标签、时长、分辨率）
  - `user_register`：用户注册事件（设备信息、注册渠道、初始兴趣标签）
- 数据格式统一为 JSON，包含 `event_time` 时间戳字段

**FR-02 数据质量校验**
- Spark Streaming 消费时对每条数据进行字段完整性校验
- 缺失关键字段（user_id, content_id, event_type）的记录打入死信队列，记录异常日志
- 数据去重：同一 `user_id + content_id + event_type` 在 60 秒窗口内仅保留首次

### 3.2 实时分析模块（Spark Streaming + MySQL）

**FR-03 实时用户画像构建**
- 按 5 分钟滑动窗口聚合用户行为特征：播放次数、完播率、点赞率、收藏率、跳过率、偏好风格/标签分布、活跃时段
- 新用户（行为数 ≤ 50）标记为冷启动用户，触发冷启动推荐策略
- 聚合结果写入 MySQL 表 `rt_user_profile`

**FR-04 实时内容热度统计**
- 按 1 分钟滚动窗口统计每首音乐/视频的：播放量、点赞数、收藏数、分享数
- 计算热度分数：`hot_score = w1×播放量 + w2×完播率 + w3×互动率`
- 结果写入 MySQL 表 `rt_content_hot`

**FR-05 实时冷启动聚类**
- 对新用户，基于注册信息（设备、渠道、初始标签）进行实时 K-Means 聚类（K 值可配，默认 8）
- 聚类结果写入 MySQL 表 `rt_coldstart_cluster`

### 3.3 离线分析模块（SparkSQL）

**FR-06 用户全量画像计算**
- 每 6 小时从 MySQL 读取 `rt_user_profile` 全量数据，执行 SparkSQL 聚合
- 产出：用户长期兴趣标签（TF-IDF 加权）、用户生命周期阶段（新用户/活跃/沉默/流失）、用户内容偏好向量（128 维）
- 结果写入 MySQL 表 `offline_user_portrait`

**FR-08 混合推荐算法**
- 冷启动用户：基于聚类群体热门内容 + 初始兴趣标签匹配 + 多样性打散（DPP 算法）
- 存量用户：协同过滤（ALS） + 内容特征回退 + 跨模态加权融合
- 推荐结果写入 MySQL 表 `offline_recommendations`

**FR-09 推荐效果指标计算**
- 每日计算：CTR、CVR、人均播放时长、人均互动次数、推荐覆盖率、推荐多样性
- 按用户群（冷启动/存量）和内容类型（音乐/视频/混合）分组统计
- 结果写入 MySQL 表 `offline_metrics`

### 3.4 后端服务模块（Python Flask）

**FR-10 用户认证**
- 注册：用户名 + 密码（bcrypt 哈希），支持邮箱验证
- 登录：JWT Token 认证，Token 有效期 2 小时，支持 Refresh Token
- 登出：服务端 Token 失效

**FR-11 权限管理（Admin）**
- 角色定义：end_user、operator、admin
- admin 可创建/禁用/删除账号，分配角色
- API 级别权限校验（装饰器拦截）

**FR-12 数据查询接口**
- `GET /api/recommendations?user_id=&page=&size=`：查询用户推荐列表
- `GET /api/metrics?start_date=&end_date=&group_by=`：查询推荐指标
- `GET /api/content/hot?type=&top_n=`：查询热门内容
- `GET /api/users/profile?user_id=`：查询用户画像
- `GET /api/coldstart/analysis`：查询冷启动用户分析数据
- 所有查询支持分页（默认 page=1, size=20）

**FR-13 数据导出**
- 导出格式：CSV / Excel
- 导出范围支持按时间、用户群、内容类型筛选

## 4. 非功能需求

### 4.1 性能需求

| 指标 | 要求 |
|------|------|
| Kafka 消费延迟 | < 500ms（P99） |
| 实时特征聚合延迟 | < 3s（端到端，从事件发生到特征写入 MySQL） |
| 离线分析完成时间 | < 2 小时（全量数据量级 ≤ 1000 万用户） |
| API 响应时间 | < 200ms（P95），< 500ms（P99） |
| 前端首屏加载 | < 2s |
| 数据大屏实时刷新 | WebSocket 推送，延迟 < 1s |
| 并发用户数 | 支持 500 并发查询 |

### 4.2 可用性需求

- 系统整体可用性 ≥ 99.5%
- Kafka 消费者支持自动重启与 offset 恢复
- MySQL 主从架构，读请求分流到从库
- 关键服务（Flask API）支持多实例部署 + Nginx 负载均衡

### 4.3 安全性需求

- 密码使用 bcrypt 哈希存储，不可逆
- JWT Token 使用 HS256 签名，密钥通过环境变量注入
- API 接口做 CORS 白名单限制
- SQL 查询使用参数化查询，防止注入
- 敏感操作（权限变更、系统配置修改）记录审计日志

### 4.4 可扩展性需求

- Kafka Topic 分区数可动态扩展
- Spark Streaming 的 batch interval 可配置
- 冷启动聚类 K 值 / 推荐策略权重可通过 Admin 后台动态调整
- MySQL 表结构支持后续字段扩展（预留 JSON 扩展字段）

### 4.5 兼容性需求

- Spark 版本：3.3.1
- Scala 版本：2.13.10
- Python 版本：3.11
- 浏览器兼容：Chrome 90+, Edge 90+, Firefox 90+

## 8. 冷启动推荐策略详述

### 8.1 策略流程

```
新用户注册 → 获取初始特征（设备、渠道、兴趣标签）
    → 实时聚类（K-Means，分配到最近群体）
    → 推荐该群体的 Top-N 热门内容（热度加权 + 时间衰减）
    → 插入少量跨群体热门内容做 Exploration（ε-greedy，ε=0.15）
    → DPP 多样性打散（避免同一歌手/标签过度集中）
    → 返回混合推荐列表（音乐:视频 = 6:4，可配置）
```

### 8.2 冷启动→存量切换

- 用户行为数 ≤ 50：冷启动策略
- 用户行为数 > 50：逐步切换至协同过滤 + 内容相似度策略
- 过渡期（行为数 30-80）：冷启动 + 协同过滤 加权混合，权重平滑过渡

## 9. 数据大屏指标定义

| 指标（中文） | 指标（英文） | 计算公式 |
|-------------|-------------|---------|
| 实时在线用户数 | Real-time Online Users | 最近 5 分钟有行为上报的去重用户数 |
| 今日推荐总数 | Daily Recommendations | 今日调用推荐接口的总次数 |
| 点击率 | CTR | 点击次数 / 曝光次数 |
| 转化率 | CVR | 完播次数 / 曝光次数 |
| 人均播放时长 | Avg Watch Duration | 总播放时长 / 去重用户数 |
| 人均互动次数 | Avg Interactions | (点赞+收藏+评论+分享) / 去重用户数 |
| 推荐覆盖率 | Coverage | 被推荐的内容数 / 总内容数 |
| 推荐多样性 | Diversity | 1 - (推荐列表内平均相似度) |
| 冷启动转化率 | Cold-start Conversion | 新用户 7 日内达到 50 次行为的比例 |

## 10. 风险与约束

| 风险/约束 | 说明 | 缓解措施 |
|----------|------|---------|
| Kafka 数据延迟 | 上游业务系统写入延迟可能导致实时分析滞后 | Spark Streaming 设置合理的 watermark，允许迟到数据（最大 30s） |
| 冷启动效果验证 | 缺乏新用户真实偏好，初期推荐可能不准 | 引入 ε-greedy 探索机制 + A/B 测试框架 |
| MySQL 写入瓶颈 | 高频实时写入可能造成 MySQL 压力 | 批量写入（batch insert），读写分离，实时表设置合理 TTL |
| 跨模态推荐难度 | 音乐与视频特征空间不同，直接融合可能降低精度 | 分模态独立训练特征向量，在排序层加权融合 |
| 单机资源限制 | 开发/测试环境资源有限 | Spark 使用 local[*] 模式，数据量使用采样模式 |
